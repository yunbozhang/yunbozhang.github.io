ApplePay

准备阶段

  1. 硬件要求：iPhone6及更新的设备，还有iPad air2和iPad mini3之后的平板，以及apple watch。具体型号（截止2016年2月）：iPhone 6，iPhone 6 Plus，iPhone 6s， iPhone 6s Plus；iPad Air 2，iPad mini 3，iPad mini 4，iPad Pro；还有Apple Watch。
  2. 系统要求：iPhone或iPad至少要升到iOS 9.2版，手表至少watch OS 2.1。
  3. 配置支付环境
    • 进入苹果开发者中心，注册App ID。App ID必须是explicit类型，否则无法使用ApplePay服务。
    • 勾选Apple Pay并点击continue。
    • 点击submit提交。
    • 在App IDs中找到刚刚注册成功的App ID，点击edit，对Apple Pay进行配置。
    • 创建MerchantID。
    • 配置Apple Pay服务，勾选Merchant ID。
    • App ID的apple pay选项变成enable表示配置成功。
    • 创建Xcode项目，填入刚注册的BundleID。
    • 在项目的target->Capabilities中，打开Apple Pay，并勾选Merchant ID。

代码实现

  1. 判断支付环境
    • 导入Passkit框架
    • 判断当前设备是否支持Apple Pay以及wallet中是否有符合所支持网络的银行卡
    • 若wallet中没有绑定所支持网络的银行卡，则点击按钮跳转到wallet中去绑定银行卡
    • 若该设备支持Apple Pay并且有绑定所支持网络的银行卡则发起支付请求
    • 支付请求配置包括国家代码、货币代码、商家ID、商家的处理方式、商家支持的网络、订单的详情，这几个要素缺一不可
    • 监听用户授权状态

AutoLayout

需求:

  1. 创建两个View(blueView,redView)
  2. blueView距上,左,右距离相等,高度50
  3. redView距blueView为20,高度与blueView相等,宽度是blueView宽度的一半
  4. redView和blueView右对齐

使用可视化实现

分析:

  1. 需要单个View分析
  2. blueView通过上,左,右距离相等,可以确定blueView的X,Y,宽;
    高度50确定blueView的高度
  3. redView距blueView为20可以确定redView的Y;
    高度与blueView相等确定redView的高度;
    宽度是blueView宽度的一半确定redView的高度;
  4. redView和blueView右对齐确定redView的X;
    代码详见附件:AutoLayoutNotCode

    使用代码实现

    核心:
    1. 如果约束的双方是父子关系添加在父类上
    2. 如果约束的双方是兄弟关系添加在父类上
    3. 如果约束是单方的添加在自身上
    4. 如果约束的双方关系复杂添加到公共的类上
    5. 使用NSLayoutConstraint实现
    6. 前提需要设置translatesAutoresizingMaskIntoConstraints为NO
      代码详见附件:AutoLayoutCode

使用简洁代码实现

使用VFL语言:

 NSArray *blueHArr = [NSLayoutConstraint    constraintsWithVisualFormat:@"H:|-20-[blueView]-20-|" 
options:kNilOptions metrics:nil views:@{@"blueView":blueView}];
  1. constraintsWithVisualFormat：这个方法返回了一个数组;
  2. @”H:|-20-[blueView(100)]-20-|”:这句话表明所添加的约束，H的意思是水平方向添加约束，blueView 视图名称，(100)代表着blueView 的宽为100，blueView距离父类的左边20，距离父类的右边是20;
  3. options：是对其方式;
  4. metrics：返回的是一个字典，它的意思是可以将约束中的值用某些字符串来代理，然后在这个字典中给多对应的字符串赋值，方便修改，并且增加可读性;

  5. views：返回的也是一个字典，这个字典的意思是：你可以将前面的view用其他字符串来代替，在这里在给他赋给对应的view的名字。

    VFL语言缺点:
    不支持乘除法,如果有乘除法仍需使用之前代码的方法
    具体代码详见附件:AutoLayoutEasyCode

自动布局小补充

  1. 如果使用VFL语言,如果有相同的值20,可以定义一个变量int a =20,然后把VFL中的20替换为a,然后在metrics中设置字典:@{@”a”:@a};
  2. 可以使用自动布局实现动画,把约束拉为插座变量,然后进行设置
    `- (void)touchesBegan:(NSSet )touches withEvent:(UIEvent )event {
    //设置左边约束
    self.left.constant = 100;
    //设置顶部约束
    self.top.constant = 200;
    [UIView animateWithDuration:2 animations:^{
    [self.view layoutIfNeeded];
    }];
    }`
  3. 系统控件会根据文字内容确定宽高

iOS 远程推送

这里只用一幅图来解释远程推送的原理：

  • 当应用第一次链接苹果 APNS 服务器的时候会上传设备的UDID 和 identifier。
  • 苹果经过加密后会生成 deviceToken 返回给客户端。
  • 客户端将返回的 deviceToken 发送给后台服务器，后台服务器存到数据库。
  • 当后台需要推送消息的时候，后台服务器发送 deviceToken 与需要推送的消息给苹果 APNS服务器，苹果 APNS 服务器通过 deviceToken 发送给客户端所推送的消息。

推送的核心就是：苹果与手机客户端保持着长链接，通过长链接能随时可以推送给手机设备消息

XML

XML优缺点

优点：

  • 格式统一，符合标准。
  • 容易与其他系统进行远程交互, 数据共享比较方便。

    缺点：

  • XML文件格式文件庞大，格式复杂， 传输占用带宽。
  • 服务器端和客户端都需要花费大量代码来解析XML，不论服务器端还是客户端都使代码变的异常复杂和不容易维护。
  • 客户端不同浏览器之间解析XML的方式不一致，需要重复编写很多代码。
  • 服务器端和客户端解析XML花费资源和时间。

JSON优缺点

优点:

  • 数据格式比较简单，易于读写，格式都是压缩的，占用带宽小。
  • 易于解析这种语言。
  • 支持多种语言，包括ActionScript，C，C#，ColdFusion，Java，JavaScript，Perl，PHP，Python，Ruby等语言服务器端语言，便于服务器端的解析。
  • 因为JSON格式能够直接为服务器端代码使用，大大简化了服务器端和客户端的代码开发量，但是完成的任务不变，且易于维护。

    缺点:

  • 没有XML格式这么推广的深入人心和使用广泛，没有XML那么通用性。
  • JSON格式目前在Web Service中推广还属于初级阶段 。
